### Step 1: Add Dynamic Typewriter Speed Calculation (D-08)

Create a helper function before the component:

```typescript
const calculateTypewriterSpeed = (
  codeLength: number,
  sceneDurationFrames: number,
): number => {
  // Reserve 20% of scene for settling/pause after reveal
  const effectiveFrames = sceneDurationFrames * 0.8;
  // Speed = chars per frame, minimum 1
  return Math.max(1, Math.ceil(codeLength / effectiveFrames));
};
```

### Step 2: Add Zoom/Pan Interpolation (D-09)

Add zoom/pan props to interface and implement interpolation:

const {
  code,
  highlightLines = [],
  typewriterSpeed: fixedSpeed,
  showLineNumbers = true,
  title,
  sceneDuration,
  zoomPanKeyframes = [], // Default: no zoom/pan

const frame = useCurrentFrame();
const { fps, durationInFrames } = useVideoConfig();

// Calculate dynamic speed if sceneDuration provided, else use fixed
const sceneFrames = sceneDuration ? Math.ceil(sceneDuration * fps) : durationInFrames;
const speed = fixedSpeed || calculateTypewriterSpeed(code.length, sceneFrames);

// Zoom/pan interpolation
const defaultScale = 1.0;
const defaultPanX = 0;
const defaultPanY = 0;

const scale = zoomPanKeyframes.length > 0
  ? interpolate(
      frame,
      zoomPanKeyframes.map(k => k.frame),
      zoomPanKeyframes.map(k => k.scale),
      { extrapolateRight: "clamp", extrapolateLeft: "clamp" }

  : defaultScale;

const panX = zoomPanKeyframes.length > 0
  ? interpolate(

  : defaultPanX;

<implementation_notes>
## Key Implementation Details

### Why Dynamic Typewriter Speed?
Fixed speed (2 chars/frame) works for short code but fails for long code (doesn't finish) or very short code (finishes too early with awkward pause). Dynamic calculation ensures code reveals at a comfortable pace within scene bounds.

### Why Zoom/Pan Instead of Scroll?
Scroll is a simple vertical movement. Zoom/pan creates a "camera through code" effect that guides viewer attention:
1. Start: Full code view (zoomed out, shows structure)
2. Zoom: Into key segment (clear, focused)
3. Pan: Move to next segment (smooth transition)

This matches Phase 3 D-07's design intent for professional code presentation.

### Why Remove CSS Transitions?
CSS `transition` is browser-timed, not frame-accurate. In Remotion:
- Same frame may render differently across renders (non-deterministic)
- Can't control exact timing
- Doesn't work well with Remotion's frame-by-frame rendering

Remotion's `interpolate` is frame-deterministic and provides exact control.

### Why Delayed Highlighting?
Immediate highlighting while typing is chaotic - viewer sees highlight flash on incomplete lines. Delayed highlighting:
1. Shows complete code first (context)
2. Then highlights key lines (focus)
This matches tutorial teaching pattern: "Here's the code, now let me explain this part..."

### Zoom/Pan Keyframe Source
For this phase, zoom/pan keyframes are optional props. If not provided, code renders normally (no zoom/pan). Future phases can add intelligent keyframe generation based on code analysis.

### Common Pitfalls to Avoid
- **DO NOT** use CSS `transition` property anywhere in Remotion components
- **DO NOT** forget `extrapolateRight: "clamp"` for `interpolate` calls
- **DO NOT** forget `extrapolateLeft: "clamp"` for bidirectional interpolation
- **DO NOT** highlight lines before code is fully revealed
- **DO NOT** use fixed typewriter speed - always calculate dynamically when sceneDuration is available
</implementation_notes>
